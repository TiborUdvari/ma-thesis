michael, if you're going to develop an applet, i think you have structured your classes slightly wrongly, as "Visualisations" should be your main class. and because of that, your problem is that you're not initializing the BApplet correctly. The least thing you should do is to call the init() method in your Visualisations class. so do this instead:
 
Code:
Visualizations v = new Visualizations();
v.init();
v.setup(); // optional
v.Initialize();

 
the init() function of BApplet will create a BGraphics instanced necessary for loadFont() used by your "Initialize()" method.

You shouldn't really need multiple BApplets. Keep one instance of BApplet and pass it as a parameter to functions that need it. I.e. instead of having a loop with drawing commands in all classes that need to draw, have a loop() in one class only and do something like the following:
 
Code:
class foo extends BApplet {
  ...
  public void loop() {
    bar.draw(this);
  }
}
 
class bar {
  public void draw(BApplet b) {
    b.ellipse(100,100, 10,10);
  }
}

 
This solves the issue of where (and when) drawing takes place, and also keeps a tidier house.

mikkel,
 
i've been using eclipse with processing for a while without using any ant build script.
 
the only thing you have to do is to create a new java project (that will hold as many sketches as you want) and copy inside bagel.jar...
 
how to make your own bagel.jar?
 
1) take a .jar file exported-to-web by processing
2) rename it to .zip
3) remove from it any class that looks like a scketch, leaving only the core processing classes (BApplet, BGraphics, etc...)
4) rename it to .jar
 
then, inside your new eclipse java project, create a new class that extends BApplet and you're ready to go...
 
note: when you first-time execute-as-an-applet inside eclipse, the size of the applet is 200x200 by default... it's possible to correct it (then it's automatically remembered) using the "run..." menu
 
finally, when you want to post your stuff to the web, be sure to recook a .jar file that will include both the processing core classes and your sketch's ones...
 
ah, one last thing: be sure that your java project is "compiling for 1.1" (for applets to work well with microsoft vm):
 
- right click your project
- go to "properties"
- then "java compiler"
- check "use project settings"
- then "compliance and classfiles"
- check that "compiler compliance level" is on 1.3 and not on 1.4
 
(on that very same panel: you have 4 checkboxes under "classfile generation", you can uncheck them all if you want your classes to be much tinier, but then you loose precious information that can be useful when debugging, i.e. error line numbers... one good alternative would be to use proguard standalone to shrink your final .jar file before posting it to the web but that's another story...)
 
hth

AFAIK - you will have to add the path to javac to your PATH variable, not to the CLASSPATH. The executable javac.exe simply has to be found by windows from wherever you call it - that's what PATH is for. CLASSPATH is the environment variable that is used by java itself to find the libraries that it uses for compilation
